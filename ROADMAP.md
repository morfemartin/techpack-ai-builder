# Roadmap

## v0.1 (shipped)
- [x] Multi-garment architecture (`src/garments/*.js` registry instead of hardcoded cap data)

## v0.2 — AI intake + flexible layout engine (in progress)
The big shift: from a **fixed** coordinate layout to an **AI-driven, flexbox-style**
tech pack generator. Two pillars, built on a hardened, secret-safe backend.

### Design system — Bauhaus, computationally semantic (`src/design/`)
- [x] Single-source design tokens (`src/design/tokens.js`) → injected CSS vars + imported by SVG builders, so UI and printed page share one palette/type/grid
- [x] **Computational color model:** color = attention role, not decoration — red = numeric index markers, blue = priority blocks (white text), yellow = small high-priority highlights (always keylined), ink/white = 80% structure
- [x] Palette tuned to stay legible in **black & white / photocopy** (distinct grayscale levels); printable surface is pure white
- [x] Two typefaces / three roles, both **system fonts** (no webfont download): Helvetica/Arial (display+ui) + native OS monospace (data); Material Symbols icons (the one true webfont - it's an icon set, no system equivalent), zero emojis, zero border-radius
- [x] Layout engine extended to resolve **percentage** basis/gap/padding — the "márgenes por porcentaje entre retículas"
- [x] Recolor the generated SVG pages with the same role tokens (the printed tech pack, not just the wizard UI)

### Phase 0 — Security & backend scaffold
- [x] Server-side proxy (`api/deepseek.js`) so the NVIDIA/DeepSeek key never reaches the browser
- [x] `SECURITY.md`, secret scanning + push protection, gitleaks in CI, Dependabot
- [ ] Rate limiting / origin check on the proxy before any public deployment

### Phase 1 — Flexbox layout engine (`src/layout/`)
- [x] Region tree model (row/column containers, grow/shrink/basis/min/max, gap, padding, align, justify) — `src/layout/builders.js` + `solve.js`
- [x] Solver that resolves the tree to absolute boxes (`solveLayout`) + a renderer that walks it to SVG (`renderLayoutToSVG`)
- [x] `buildPage1` (parts spec sheet + 4-view diagram) rebuilt on the engine — part-table rows are now `leaf({ grow: 1, min: 16 })` instead of a hand-derived `Math.floor(bodyH / partsCount)` formula. Verified geometric parity against the old fixed-coordinate output in `src/pages/buildPages.test.js`, and fixed a real bug surfaced by the refactor: the 4-view grid used to start flush with the DETAILS bar's own top edge instead of below it, so its opaque background silently covered most of the "DETAILS" label.
- [ ] `buildDesignPage` still uses hand-computed pixel math — port it to the engine next (same pattern as `buildPage1`)
- [x] Unit tests for the solver (`src/layout/solve.test.js`, 16 cases: grow/shrink/clamping/nesting/justify/align/data-volume flexing)

### Phase 3.5 — Systemic reasoning core (shared by every intake door)
- [x] **F3.1 — Requirements engine + skip-the-obvious + numbered options** (`src/core/techpackRequirements.js`): the shared "tech-pack reasoning core". `analyzeRequirements()` makes ONE DeepSeek call that reasons like a textile technician about what a factory needs for a given garment, splitting fields into `assumed` (standard/obvious - pre-filled, never asked), `known` (from the seed), and `ask` (genuinely decisive - each with 2-4 short numbered options). The chat (`GarmentChat.jsx`) is now a phase-aware WALKER: it walks the `ask` fields locally, one at a time, with clickable numbered option chips, instead of a fresh DeepSeek call every turn - bounded and cheap at runtime no matter how long the intake gets. Pure walker helpers (`pendingFields`/`applyAnswer`/`isComplete`/`reqsToParts`) are unit-tested; the test file was drafted by DeepSeek (orchestrated via the `deepseek-code` skill) and reviewed/integrated. Verified live: "Camisa formal de vestir" → the model assumed the obvious (hem finish, no lining) and asked only the 6 decisive fields (fabric, collar type, collar stiffness, front placket, cuff type, fit, back yoke) with numbered options; answering them produced a real factory-grade 9-row parts sheet in Vista Previa. Also fixed live: the first prompt was so thorough it truncated at the token cap (`finish_reason: length`) - constrained it to the 6-10 most decisive fields with short `why`/options.
- [x] **F3.1b — real streaming progress for the requirements analysis**: user feedback on F3.1 - the "Pensando..." wait felt opaque, and they explicitly wanted to genuinely "see what the model is thinking," not a fake canned progress bar. `api/deepseek.js` now proxies NVIDIA's `stream: true` SSE responses through unchanged (`res.write()` piping, checking status before touching the body so non-2xx failures still degrade like the non-streaming path); `deepseekClient.js` gained `deepseekChatStream()` (hand-rolled SSE parsing, no new deps, reuses the existing retry/backoff for the "open the stream" phase only - once bytes are flowing there's no retry). Considered showing the raw `reasoning_content` chain-of-thought (`thinking: true`) verbatim, but it streams out in English regardless of the Spanish conversation and is unpolished internal scratchpad text - rejected in favor of deriving a genuine, non-fabricated status line from the actual streamed JSON: `extractLastCompletedLabel()` only trusts a field's label once its whole object closes (anchored on `"why"`, the last key in the shape), so `GarmentChat.jsx`'s new progress bar shows real field names as they finish generating (e.g. "Analizando: Composición de fibra…" → "Tipo de manga…"), with percent estimated from SSE-event count against a calibrated budget. Verified live end-to-end through the full local pipe (browser → Vite proxy → dev shim → `api/deepseek.js` → NVIDIA): the bar advanced with genuinely different real labels across consecutive screenshots seconds apart, and the final walked result (assumed/known/ask fields, numbered options) was unchanged from the non-streaming version.
- [x] **F3.2 — design-level pass: real pages per design element** (`src/core/techpackRequirements.js`): a second DeepSeek call, `analyzeDesignExpression()`, runs once the general construction fields (F3.1) are done - reasons like a technical designer about which DISCRETE elements of the garment (embroidered logo, personalized hardware, printed/woven label, etc.) need their own design page, given the general answers as context.
