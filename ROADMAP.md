## v0.2 ‚Äî AI intake + flexible layout engine (in progress)
The big shift: from a **fixed** coordinate layout to an **AI-driven, flexbox-style**
tech pack generator. Two pillars, built on a hardened, secret-safe backend.

### Design system ‚Äî Bauhaus, computationally semantic (`src/design/`)
- [x] Single-source design tokens (`src/design/tokens.js`) ‚Üí injected CSS vars + imported by SVG builders, so UI and printed page share one palette/type/grid
- [x] **Computational color model:** color = attention role, not decoration ‚Äî red = numeric index markers, blue = priority blocks (white text), yellow = small high-priority highlights (always keylined), ink/white = 80% structure
- [x] Palette tuned to stay legible in **black & white / photocopy** (distinct grayscale levels); printable surface is pure white
- [x] Two typefaces / three roles, both **system fonts** (no webfont download): Helvetica/Arial (display+ui) + native OS monospace (data); Material Symbols icons (the one true webfont - it's an icon set, no system equivalent), zero emojis, zero border-radius
- [x] Layout engine extended to resolve **percentage** basis/gap/padding ‚Äî the "m√°rgenes por porcentaje entre ret√≠culas"
- [x] Recolor the generated SVG pages with the same role tokens (the printed tech pack, not just the wizard UI)

### Phase 0 ‚Äî Security & backend scaffold
- [x] Server-side proxy (`api/deepseek.js`) so the NVIDIA/DeepSeek key never reaches the browser
- [x] `SECURITY.md`, secret scanning + push protection, gitleaks in CI, Dependabot
- [ ] Rate limiting / origin check on the proxy before any public deployment

### Phase 1 ‚Äî Flexbox layout engine (`src/layout/`)
- [x] Region tree model (row/column containers, grow/shrink/basis/min/max, gap, padding, align, justify) ‚Äî `src/layout/builders.js` + `solve.js`
- [x] Solver that resolves the tree to absolute boxes (`solveLayout`) + a renderer that walks it to SVG (`renderLayoutToSVG`)
- [x] `buildPage1` (parts spec sheet + 4-view diagram) rebuilt on the engine ‚Äî part-table rows are now `leaf({ grow: 1, min: 16 })` instead of a hand-derived `Math.floor(bodyH / partsCount)` formula. Verified geometric parity against the old fixed-coordinate output in `src/pages/buildPages.test.js`, and fixed a real bug surfaced by the refactor: the 4-view grid used to start flush with the DETAILS bar's own top edge instead of below it, so its opaque background silently covered most of the "DETAILS" label.
- [ ] `buildDesignPage` still uses hand-computed pixel math ‚Äî port it to the engine next (same pattern as `buildPage1`)
- [x] Unit tests for the solver (`src/layout/solve.test.js`, 16 cases: grow/shrink/clamping/nesting/justify/align/data-volume flexing)

### Phase 2 ‚Äî AI intake (DeepSeek)
- [x] **Fase A ‚Äî DeepSeek client foundation** (`src/core/deepseekClient.js`): `deepseekChat`/`extractStructured` against the proxy, with automatic retry on the NVIDIA free-tier's frequent transient `ResourceExhausted` 503s. Local dev gap closed: `api/deepseek.js` is a Vercel function that plain `vite dev` never executes - `npm run dev` now runs `scripts/dev.mjs`, a small local shim (same handler code, no Vercel account needed) proxied through Vite (`vite.config.js`). Verified with real end-to-end calls against the live NVIDIA endpoint, not just mocks.
- [x] **Fase B ‚Äî CSV import** (registered garments) (`src/core/csvImport.js`): upload one CSV in the Piezas step, DeepSeek (not a rigid column parser - people fill these out however makes sense to them) extracts `{parts, designs}` and pre-fills the existing Piezas/Dise√±os steps for review. Matched part labels overlay onto the garment's canonical id/order (so the result is always complete even from a partial CSV); unrecognized rows become custom parts instead of being dropped. Verified live with a deliberately messy, non-tabular CSV (mixed `Etiqueta: valor` and comma-separated sections) - correctly extracted and mapped.
- [x] **Fase C ‚Äî "Prenda desde 0"**: a DeepSeek chat builds a garment definition question by question (parts, positions, design/embroidery intent), generic-silhouette fallback in `buildPages.js`/`Preview.jsx` for garments with no hand-drawn `guides`/`callouts`, then the existing Dise√±os step for image+dimension upload. Draft is downloadable as a `garments/<id>.js` scaffold (placeholder silhouette, `// TODO` pointing at CONTRIBUTING.md) so it can be PR'd in as an officially-supported garment. Verified live end-to-end with a real DeepSeek conversation (Polo: cuello cl√°sico con 3 botones n√°car, mangas cortas con ribete, bajo cola de pato, logo bordado PANTONE 286C) through to Vista Previa (generic-silhouette rectangles, numbered part chips, no invented callouts) and a working "Descargar prenda (.js)" download (`custom-polo.js`). Also found and fixed live: a React lazy-initializer bug that crashed the wizard when switching to a custom garment, and added graceful degradation for turns where the model replies in plain text instead of the JSON envelope.
- [x] **Fase D ‚Äî CSV/Wilcom import gets embroidery specs + companion photos** (`src/core/csvImport.js`): when the uploaded CSV describes embroidery digitizing data (stitches, color changes, stops/trims, stabilizer top/backing, thread/bobbin - the kind of sheet Wilcom exports), DeepSeek now also returns an `emb{}` object per design using the exact `EMPTY_EMB` field names, with an explicit prompt rule forcing `tec` to an exact recognized embroidery label so the `isEmbTec` gate reliably fires. A new optional multi-photo upload sits next to the CSV input in the Piezas step; `matchImagesToDesigns()` attaches photos via a DeepSeek-returned filename hint first (the model is text-only here - it never sees pixel data, only filenames), falling back to order-based pairing for anything left over, and surfaces any unmatched photos instead of dropping them silently. Verified live with a synthetic Wilcom-style CSV (12500 stitches, 4 color changes, 2 stops, 1 trim, soluble topping, cut-away backing, Rayon/bobbin thread) referencing a photo by filename - the extracted `emb` data and matched photo landed correctly in the existing Dise√±os step's `EmbForm`/`ImageUploader`, with zero new rendering code needed.
- [x] **Fase E ‚Äî "Prenda desde 0" chat produces structured designs, not a discarded free-text summary**: fixed a real bug found in live use - the chat's `draft.designs` used to be one free-text string that `handleGarmentChatComplete` never even read, so anything said about custom buttons, embroidery, or Drive links silently never became a real page. `draft.designs` is now a structured array (`name/pos/posDetail/tec/driveLink/notes`), with an explicit prompt rule for when something gets its own page (has its own art, a Drive link, or an embroidery spec) vs. staying a flat `parts[]` attribute, plus a rule to probe construction-detail variants (cuffs or not, straight hem or shirttail, drawstring or not) before considering a piece complete. `GarmentChat` now takes a `tecs` prop (same list `csvImport.js` already uses) so `tec` lands on an exact recognized value. New `mapChatDesignsToDesigns()` (`buildCustomGarment.js`) converts the chat's designs into the same partial shape CSV import already produces, reusing `Object.assign(newDesign(), ...)` - zero new UI, the existing Dise√±os step just receives real pre-filled cards instead of one blank default. Verified live end-to-end with a real DeepSeek conversation (Polo: cuello con pu√±o azul marino - a flat attribute; custom gold buttons with a Drive link - its own design; embroidered logo PANTONE 286C, `tec` landing on the exact "Bordado 3D" - its own design with the embroidery block firing correctly): the flat attribute stayed in Piezas, both customized elements appeared as separate entries in the live "Borrador" panel, Dise√±os arrived with both cards pre-filled (not one blank), and the generated tech pack showed 1 main page + 2 design pages - the concrete proof the layout responds to what's described, not just the data.

### Phase 3.5 ‚Äî Systemic reasoning core (shared by every intake door)
- [x] **F3.1 ‚Äî Requirements engine + skip-the-obvious + numbered options** (`src/core/techpackRequirements.js`): the shared "tech-pack reasoning core". `analyzeRequirements()` makes ONE DeepSeek call that reasons like a textile technician about what a factory needs for a given garment, splitting fields into `assumed` (standard/obvious - pre-filled, never asked), `known` (from the seed), and `ask` (genuinely decisive - each with 2-4 short numbered options). The chat (`GarmentChat.jsx`) is now a phase-aware WALKER: it walks the `ask` fields locally, one at a time, with clickable numbered option chips, instead of a fresh DeepSeek call every turn - bounded and cheap at runtime no matter how long the intake gets. Pure walker helpers (`pendingFields`/`applyAnswer`/`isComplete`/`reqsToParts`) are unit-tested; the test file was drafted by DeepSeek (orchestrated via the `deepseek-code` skill) and reviewed/integrated. Verified live: "Camisa formal de vestir" ‚Üí the model assumed the obvious (hem finish, no lining) and asked only the 6 decisive fields (fabric, collar type, collar stiffness, front placket, cuff type, fit, back yoke) with numbered options; answering them produced a real factory-grade 9-row parts sheet in Vista Previa. Also fixed live: the first prompt was so thorough it truncated at the token cap (`finish_reason: length`) - constrained it to the 6-10 most decisive fields with short `why`/options.
- [x] **F3.1b ‚Äî real streaming progress for the requirements analysis**: user feedback on F3.1 - the "Pensando..." wait felt opaque, and they explicitly wanted to genuinely "see what the model is thinking," not a fake canned progress bar. `api/deepseek.js` now proxies NVIDIA's `stream: true` SSE responses through unchanged (`res.write()` piping, checking status before touching the body so non-2xx failures still degrade like the non-streaming path); `deepseekClient.js` gained `deepseekChatStream()` (hand-rolled SSE parsing, no new deps, reuses the existing retry/backoff for the "open the stream" phase only - once bytes are flowing there's no retry). Considered showing the raw `reasoning_content` chain-of-thought (`thinking: true`) verbatim, but it streams out in English regardless of the Spanish conversation and is unpolished internal scratchpad text - rejected in favor of deriving a genuine, non-fabricated status line from the actual streamed JSON: `extractLastCompletedLabel()` only trusts a field's label once its whole object closes (anchored on `"why"`, the last key in the shape), so `GarmentChat.jsx`'s new progress bar shows real field names as they finish generating (e.g. "Analizando: Composici√≥n de fibra‚Ä¶" ‚Üí "Tipo de manga‚Ä¶"), with percent estimated from SSE-event count against a calibrated budget. Verified live end-to-end through the full local pipe (browser ‚Üí Vite proxy ‚Üí dev shim ‚Üí `api/deepseek.js` ‚Üí NVIDIA): the bar advanced with genuinely different real labels across consecutive screenshots seconds apart, and the final walked result (assumed/known/ask fields, numbered options) was unchanged from the non-streaming version.
- [x] **F3.2 ‚Äî design-level pass: real pages per design element** (`src/core/techpackRequirements.js`): a second DeepSeek call, `analyzeDesignExpression()`, runs once the general construction fields (F3.1) are done - reasons like a technical designer about which DISCRETE elements of the garment (embroidered logo, personalized hardware, printed/woven label, etc.) need their own design page, given the general answers as context. Each element is a group of 2-4 fields sharing a `designSlot` id, tagged with `designField` (`name`/`position`/`technique`/`driveLink`/`detail`) - reuses the exact same field shape/normalizer/walker (`pendingFields`/`applyAnswer`/`isComplete`) as F3.1's general pass, just tagged `category: "design"`, so `GarmentChat.jsx` needed zero new walking logic, only a new `"designAnalyzing"`/`"designing"` phase pair mirroring the existing `"analyzing"`/`"asking"` ones (including the same streaming progress bar). New `mergeDesignFields()` folds the design fields into the existing `reqs`; new `reqsToDesigns()` groups completed design fields by `designSlot` back into the same design-object shape `mapChatDesignsToDesigns()` (Fase E) already expects - zero changes needed downstream. Delegated to DeepSeek (direct NVIDIA API calls, bypassing a flaky `ai-orchestrator` HTTP/2 issue - forcing `--http1.1` on curl fixed it) for the full first draft of `analyzeDesignExpression`/`mergeDesignFields`/`reqsToDesigns` plus their tests, given the exact contract as spec; reviewed and integrated, with one real bug caught and fixed: `normalizeRequirements()` was silently dropping the new `designSlot`/`designField` keys, which would have made `reqsToDesigns()` never match anything. Verified live end-to-end with a real DeepSeek conversation for a Polo: the model correctly identified three distinct design elements (embroidered chest logo, personalized buttons, woven brand label), asked through all of them with numbered options, and the generated tech pack showed exactly 1 main page + 3 design pages. Observed live (not a bug, model-output variance worth noting): the model sometimes represents a location as a `"detail"` field instead of using the dedicated `"position"` designField - the data still lands safely in `notes`/`posDetail` rather than being lost, just not always in the ideal UI slot; a candidate for future prompt refinement.
